Trivy Security Scan Workflow Changelog

Latest Changes

2025-04-25: Updated Trivy Security Scan Workflow

The Trivy Security Scan workflow has been updated to improve the security scanning capabilities of the repository. This update simplifies the workflow and improves its effectiveness.

Key Changes

  • Changed permissions: Modified the permissions to be more restrictive (removed contents: write and actions: read) while maintaining the necessary permissions for security scanning
  • Updated checkout action: Changed to use actions/checkout@v3 with recursive submodule initialization
  • Direct installation: Now directly installs Trivy on the runner instead of using pre-built actions
  • Streamlined scanning: Simplified the scanning process with more direct commands
  • Exit code changes: Modified to fail the workflow when vulnerabilities are found (exit code 1), improving security enforcement
  • Removed report generation: Removed the report generation and commit functionality to focus on security enforcement
  • Simplified steps: Reduced the number of steps from 10 to 6, making the workflow more maintainable

Benefits

  • More focused security enforcement
  • Clearer failure conditions for security issues
  • Better submodule support
  • Streamlined workflow with fewer external dependencies
  • More direct control over the Trivy installation and scanning options

Implementation Details

The new workflow:

  • Installs Trivy directly on the runner
  • Scans for both high and critical vulnerabilities
  • Scans specifically for secrets
  • Has a separate step for dependency vulnerabilities
  • Includes a step to handle Trivy warnings when failures occur

These changes align with best practices for security scanning in CI/CD pipelines.
