AI Lessons Learned - Material UI to Nebular Migration

Component Migration Patterns

1. Layout Component Migration

When migrating from Material's layout components to Nebular's, we discovered several key patterns:

  • Nebular's layout system is more opinionated and requires proper nesting of components
  • The nb-layout component must be the root container
  • Sidebar and header components need specific placement within the layout hierarchy
Best Practice: Always start migration from the root layout components and work your way down to ensure proper component hierarchy.

2. Form Control Differences

Key differences in form control implementation between Material and Nebular:

  • Nebular form controls use different validation patterns
  • Status classes are handled differently in Nebular
  • Form field appearance customization requires different approaches
// Material UI
<mat-form-field appearance="outline">
  <input matInput [formControl]="control">
  <mat-error *ngIf="control.hasError('required')">
    Required field
  </mat-error>
</mat-form-field>

// Nebular
<nb-form-field>
  <input nbInput [formControl]="control">
  <ng-container *ngIf="control.hasError('required')">
    Required field
  </ng-container>
</nb-form-field>

3. Theme Migration Strategy

Lessons learned about theme migration:

  • Nebular's theme system is more extensive and requires careful planning
  • Custom variables need to be mapped to Nebular's theme structure
  • Some Material-specific color functions need to be reimplemented
// Material UI
$theme-colors: (
  primary: mat-palette($mat-indigo),
  accent: mat-palette($mat-pink)
);

// Nebular
$nb-themes: nb-register-theme((
  color-primary-100: #f2f6ff,
  color-primary-500: #3366ff,
  color-primary-900: #091c7a
), default, default);

Common Challenges and Solutions

1. Icon System Migration

Challenges faced when migrating from Material Icons to Eva Icons:

  • Icon name mapping is not one-to-one
  • Eva Icons have different naming conventions
  • Some icons need to be combined to achieve the same effect
Solution: Created a mapping service to handle icon name translations and fallbacks.

2. Dialog System Differences

Key differences in dialog implementation:

  • Nebular uses a different context system
  • Dialog styling and animations differ significantly
  • Dialog results are handled differently
// Material UI
const dialogRef = this.dialog.open(DialogComponent, {
  data: { ... }
});

// Nebular
const dialogRef = this.dialogService.open(DialogComponent, {
  context: { ... }
});

3. Performance Considerations

Performance impacts observed during migration:

  • Nebular's theme system can impact initial load time
  • Dynamic theme switching requires optimization
  • Some components need lazy loading strategies
Solution: Implemented lazy loading for feature modules and optimized theme variable usage.
