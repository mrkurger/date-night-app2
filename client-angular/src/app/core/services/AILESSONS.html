AI Lessons: Services

This file contains lessons learned and best practices for the Services component/module.

Interface Consistency in Tests

When working with complex interfaces like Ad and Favorite, it's crucial to ensure that mock objects in tests fully implement the required interface properties. Missing properties can lead to subtle type errors that may not be immediately apparent.

Problem Encountered

In the FavoriteService tests, mock objects were missing required properties from the Ad interface, causing type errors.

Solution

Always create complete mock objects that satisfy all required properties of the interface: 

// Complete mock object
const mockAd = {
  _id: 'ad1',
  title: 'Test Ad',
  userId: 'user1',
  isActive: true,
  isFeatured: false,
  isTrending: false,
  isTouring: false,
  viewCount: 0,
  clickCount: 0,
  inquiryCount: 0,
  createdAt: new Date().toISOString(),
  updatedAt: new Date().toISOString(),
  location: {
    city: 'Test City',
    county: 'Test County'
  }
};
Tip: Consider creating factory functions for generating test data to ensure consistency across tests.

Observable Properties vs. Methods in Tests

When testing components that use services with observable streams, it's often better to mock the observable properties directly rather than methods that return observables.

Problem Encountered

In the chat component tests, we were mocking methods that return observables:

chatServiceSpy.onNewMessage.and.returnValue(newMessageSubject.asObservable());
chatServiceSpy.onMessageRead.and.returnValue(messageReadSubject.asObservable());
chatServiceSpy.onTypingIndicator.and.returnValue(typingIndicatorSubject.asObservable());

Solution

A more maintainable approach is to directly set the observable properties:

// Set up the socket event handlers as properties instead of methods
chatServiceSpy.newMessage$ = newMessageSubject.asObservable();
chatServiceSpy.messageRead$ = messageReadSubject.asObservable();
chatServiceSpy.typingIndicator$ = typingIndicatorSubject.asObservable();

This approach better reflects the actual service implementation and makes the tests more resilient to implementation changes.

Specific vs. Generic Assertions

When testing complex objects, using specific assertions for critical properties rather than exact equality checks can make tests more maintainable.

Problem Encountered

In the FavoriteService tests, we were using exact equality checks for complex objects: 

service.getFavorites().subscribe(favorites => {
  expect(favorites).toEqual(mockFavorites);
});

Solution

A more maintainable approach is to assert on specific properties that are critical to the test: 

service.getFavorites().subscribe(favorites => {
  expect(favorites.length).toEqual(mockFavorites.length);
  expect(favorites[0]._id).toEqual(mockFavorites[0]._id);
  expect(favorites[1]._id).toEqual(mockFavorites[1]._id);
});

This approach makes tests less brittle when the object structure changes in ways that don't affect the core functionality being tested.

CSS Integration Strategies

When integrating external CSS frameworks like Emerald UI, it's important to choose the right integration strategy to avoid conflicts with existing styles.

Approach Used

We integrated Emerald UI CSS by importing it in component-specific SCSS files rather than globally: 

@use '../../../styles/design-system/index.scss' as ds;
/* Removed emerald-ui import as it's causing build issues */

This approach allows for more granular control over which components use the Emerald UI styles, reducing the risk of style conflicts.

Benefits

  • Component-specific styling control
  • Reduced risk of global style conflicts
  • Easier to remove or replace the framework in the future
  • Better performance as styles are only loaded where needed

Modern SASS Practices

As SASS evolves, certain functions and practices become deprecated. It's important to stay current with modern SASS practices to ensure compatibility with future versions.

Problem Encountered

We encountered several SASS deprecation warnings related to the use of functions like darken() and lighten(), as well as parent selector nesting issues. 

// Old approach with deprecated functions
&:hover:not(:disabled) {
  background-color: darken(ds.$color-primary, 5%);
}

Solution

We updated our SCSS files to use the newer color.scale() function and proper parent selector syntax: 

// Import the color module
@use 'sass:color';

// Modern approach with color.scale()
&:hover:not(:disabled) {
  background-color: color.scale(ds.$color-primary, $lightness: -5%);
}

Benefits

  • Future-proof code that works with newer SASS versions
  • More explicit and readable color manipulations
  • Better compatibility with SASS modules system
  • Elimination of deprecation warnings

File Encryption Implementation

When implementing file encryption in a web application, it's important to leverage the browser's native Web Crypto API for secure operations.

Approach Used

We created a dedicated FileEncryptionService that builds on top of our existing EncryptionService to handle file-specific encryption needs: 

@Injectable({
  providedIn: 'root',
})
export class FileEncryptionService {
  constructor(private encryptionService: EncryptionService) {}

  async encryptFile(file: File): Promise<{ encryptedData: ArrayBuffer; iv: Uint8Array }> {
    const fileBuffer = await file.arrayBuffer();
    const iv = crypto.getRandomValues(new Uint8Array(12));
    const encryptedData = await this.encryptionService.encrypt(fileBuffer, iv);

    return { encryptedData, iv };
  }

  async decryptFile(encryptedData: ArrayBuffer, iv: Uint8Array): Promise {
    return this.encryptionService.decrypt(encryptedData, iv);
  }
}

Key Learnings

  • Use File.arrayBuffer() to efficiently convert files to binary data
  • Generate cryptographically secure random values for the initialization vector (IV)
  • Separate file-specific encryption logic from the core encryption service
  • Return both the encrypted data and the IV, as both are needed for decryption
  • Use async/await for cleaner handling of Promise-based crypto operations

Testing Considerations

When testing encryption services, it's important to:

  • Mock the underlying crypto operations to avoid actual encryption in tests
  • Test the full encryption and decryption cycle to ensure data integrity
  • Verify that the service correctly handles different file types and sizes
  • Check error handling for cases like invalid IVs or corrupted data
