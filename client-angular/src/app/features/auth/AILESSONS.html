AI Lessons: Auth

This file contains lessons learned and best practices for the Auth component/module.

Best Practices

  • Always use HTTP-only cookies for storing refresh tokens to prevent XSS attacks
  • Implement short-lived access tokens (15 minutes) with refresh token rotation for better security
  • Use bcrypt for password hashing with a salt factor of at least 10
  • Implement rate limiting on authentication endpoints to prevent brute force attacks
  • Always validate user input on both client and server sides
  • Use HTTPS for all authentication-related communication
  • Use standalone Angular components for authentication to minimize bundle size and improve loading performance
  • Implement OAuth 2.0 with PKCE for social login providers to enhance security in public clients

Common Patterns

  • JWT-based authentication with refresh token rotation
  • Interceptor pattern for automatically adding auth tokens to API requests
  • Guard pattern for protecting routes based on authentication state
  • Reactive forms with validation for authentication forms
  • Observable-based authentication state management
  • Social authentication integration with OAuth providers

Known Issues

  • Token expiration handling can cause race conditions when multiple requests fail simultaneously
  • Social authentication may fail in private browsing mode due to third-party cookie restrictions
  • Password reset flow may break if emails are delayed or caught by spam filters
  • Session management across multiple devices can be confusing for users

Implementation Notes

  • Authentication state is managed using a BehaviorSubject in the AuthService
  • JWT tokens are stored in localStorage for access tokens and HTTP-only cookies for refresh tokens
  • Token refresh is handled automatically by the AuthInterceptor when a 401 response is received
  • Social authentication uses the OAuth 2.0 authorization code flow with PKCE for better security
  • Password reset tokens are short-lived (1 hour) and single-use

OAuth Integration

When implementing OAuth authentication with multiple providers (Google, Facebook, Apple, Microsoft), we've learned several important lessons:

Configuration

  • Client IDs: Each OAuth provider requires a unique client ID obtained from their developer console. These should be stored in environment variables, not hardcoded.
  • Redirect URIs: Must be consistently configured both in the OAuth provider console and the application code. They must exactly match.
  • Scopes: Request only the minimum scopes needed (typically 'email' and 'profile') to comply with privacy regulations.

Implementation

  • Universal Callback Component: Using a single auth-callback component that handles responses from all providers reduces code duplication.
  • Provider Detection: The callback should detect which provider initiated the OAuth flow, typically via query parameters.
  • Error Handling: OAuth callbacks should gracefully handle authentication failures with clear user feedback.

Token Management

  • Token Storage: OAuth tokens should be handled securely, preferably using HttpOnly cookies set by the server, not localStorage.
  • Token Refresh: Implement automatic refresh for providers that support refresh tokens to maintain sessions without requiring re-authentication.
  • User Mapping: Create a consistent user profile structure regardless of which OAuth provider was used, mapping provider-specific fields to your application's user model.

UI/UX Considerations

  • Provider Buttons: Follow each provider's branding guidelines for logo usage and button styling.
  • Loading States: Show clear loading indicators during the OAuth process, which may take several seconds.
  • Account Linking: Consider how to handle cases where a user tries to login with a social account that matches an existing email address.
